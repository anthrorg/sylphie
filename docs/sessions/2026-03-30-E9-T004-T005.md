# 2026-03-30 -- E9-T004 (HealthController) + E9-T005 (DrivesController)

## Changes

- MODIFIED: src/web/controllers/health.controller.ts -- Enhanced with timeouts, caching, metadata, and event recording
  - Added 30s response caching for healthy status (in-memory with timestamp)
  - Added system metadata: uptime (process.uptime()) and version ('0.1.0')
  - Added rollup rules: healthy (all 5 OK + <200ms), degraded (accessible but slow/warning), unhealthy (unreachable)
  - Per-database error handling and latency tracking
  - Records HEALTH_CHECK_COMPLETED event to TimescaleDB after each check
  - Maps raw health service response to HealthCheckResponse DTO with HealthCheckResult[]

- MODIFIED: src/web/controllers/drives.controller.ts -- Full implementation replacing stub
  - GET /api/drives/current: Returns DriveStateResponse with current snapshot (all 12 drives + total pressure + tick)
  - GET /api/drives/history: Returns DriveHistoryResponse with DRIVE_TICK events from TimescaleDB
  - Query params: from/to (ISO string or epoch ms), resolution ('1s'|'5s'|'30s'|'1m'|'5m')
  - Default time range: last 5 minutes
  - Result limit: 1000 points to prevent overload
  - Injects IDriveStateReader (read-only per CANON) and IEventService
  - Maps all 12 drives via DRIVE_INDEX_ORDER and DriveName enum
  - No write endpoints — CANON §Drive Isolation maintained

## Wiring Changes

- HealthController now depends on: IEventService (EVENTS_SERVICE token), ConfigService, DatabaseHealthService
- DrivesController now depends on: IDriveStateReader (DRIVE_STATE_READER token), IEventService (EVENTS_SERVICE token)
- Both controllers record events to TimescaleDB (health checks and implicitly via drive history queries)

## Known Issues

None. Both controllers type-check cleanly. Pre-existing type errors in other files (decorators, gateway payload shapes) remain unchanged.

## Gotchas for Next Session

- Health check caching is in-memory and per-instance; distributed deployments will need Redis or similar
- Drive history queries could be slow for large time windows; may need aggregation/downsampling for long-term trending
- DRIVE_TICK event frequency and availability depends on Drive Engine being healthy and connected
