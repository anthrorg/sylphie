# 2026-03-30 — E9-T008 & E9-T012: MetricsController and VoiceController

## Changes
- NEW: `src/web/controllers/metrics.controller.ts` — Complete implementation of all 7 CANON health metrics endpoints
  - GET /api/metrics/health — Aggregates Type1/Type2 ratio, prediction MAE, provenance distribution, behavioral diversity, guardian response rate, interoceptive accuracy, drive resolution times
  - GET /api/metrics/type-ratio?window= — Time-windowed Type 1/Type 2 ratio
  - GET /api/metrics/predictions?window= — Time-windowed prediction MAE
  - GET /api/metrics/provenance — Current provenance distribution from WKG
  - DevelopmentGuard applied to entire controller
  - Metrics computation methods with ACT-R confidence dynamics

- NEW: `src/web/controllers/voice.controller.ts` — Complete STT/TTS HTTP endpoints
  - POST /api/voice/transcribe — Audio-to-text via OpenAI Whisper, graceful degradation on STT failure
  - POST /api/voice/synthesize — Text-to-audio via OpenAI TTS with JSON fallback on TTS failure
  - Fire-and-forget event recording for VOICE_TRANSCRIPTION_COMPLETED and VOICE_SYNTHESIS_COMPLETED
  - Never blocks on audio service failure per CANON communication rules
  - Includes simple UUID v4 generation to avoid external dependencies

## Wiring Changes
- Both controllers injected with required services (IEventService, IWkgService, ISelfKgService, IDriveStateReader, ISttService, ITtsService)
- MetricsController guarded by DevelopmentGuard
- Both registered in WebModule (previously configured)

## Known Issues
- Trend analysis in MetricsController is placeholder (always returns 'stable') — full implementation requires historical window tracking
- Interoceptive accuracy uses simplified Self KG lookup — full version would integrate with KG(Self) self-evaluation nodes
- Drive resolution time computation uses simplified event payload parsing — future versions should use stronger event typing

## Gotchas for Next Session
- uuid package not in dependencies — used simple v4 generation in VoiceController instead
- DriveSnapshot structure requires pressureVector (PressureVector record), not simple drives array
- IWkgService method is queryGraphStats() not graphStats()
- IDriveStateReader.getCurrentState() is synchronous, not async
- RawBodyRequest decorator not available in standard NestJS — used @Req() with Buffer body instead
