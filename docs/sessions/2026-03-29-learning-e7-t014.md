## Changes

### NEW: JobRegistryService (211 lines)
- **File:** `src/learning/jobs/job-registry.service.ts`
- **Purpose:** Registry and orchestrator of learning jobs executed during consolidation cycles
- **Key Features:**
  - Registers all 5 learning jobs in strict dependency order
  - Enforces dependency chain: TemporalPatternJob, CorrectionProcessingJob (base), then ProcedureFormationJob, SentenceProcessingJob (depend on TemporalPattern), finally PatternGeneralizationJob
  - Executes jobs sequentially with per-job error isolation (one failure doesn't kill cycle)
  - Returns aggregated JobResult[] with artifact counts, latency, and issues
  - Logs execution metrics per job
  - No external events emitted (job results aggregated into CONSOLIDATION_CYCLE_COMPLETED)

### NEW: ConsolidationService (374 lines)
- **File:** `src/learning/consolidation/consolidation.service.ts`
- **Purpose:** Orchestrates full consolidation pipeline for event batches
- **Key Features:**
  - `selectBatch()`: Queries learnable events, ranks by salience via EventRankerService, selects top N (max 5 per CANON)
  - `consolidate()`: Executes full pipeline for each event:
    * Stage 1: Entity extraction via IEntityExtractionService
    * Stage 2: Edge refinement via IEdgeRefinementService
    * Stage 3: Contradiction detection via IContradictionDetector
    * Stage 4: WKG upsert (non-contradicting entities/edges)
  - Maps ContradictionCheckResult resolution types to Contradiction types
  - Emits CONTRADICTION_DETECTED events to TimescaleDB with drive snapshot
  - Returns ConsolidationResult with metrics, entities, edges, contradictions, job results
  - Injects DRIVE_STATE_READER to obtain drive snapshot for event emission
  - Per-event error isolation prevents batch failures

### NEW: LearningMetricsService (149 lines)
- **File:** `src/learning/metrics/learning-metrics.service.ts`
- **Purpose:** Tracks and aggregates learning cycle metrics
- **Key Features:**
  - Maintains in-memory rolling window of 100 recent cycle metrics
  - `recordCycleMetrics()`: Records metrics, updates aggregate stats
  - `getRecentMetrics(limit)`: Returns N most recent cycles (default 10)
  - `getCycleDurationAverage()`: Returns average duration across all cycles
  - `getTotalCycles()`: Returns total cycle count
  - `getAggregateStats()`: Diagnostics object with totals and per-cycle averages
  - Used by LearningService and MaintenanceCycleService for diagnostics

## Gotchas for Next Session

1. **Job Results Aggregation:** Job results are NOT emitted as individual events. They're aggregated into CONSOLIDATION_CYCLE_COMPLETED payload. If detailed job tracking becomes critical, extend CONSOLIDATION_CYCLE_COMPLETED payload or create new event type in EventType union.

2. **Contradiction Resolution Mapping:** The mapping from ContradictionCheckResult.resolution (GUARDIAN_REVIEW | SUPERSEDED | COEXIST) to Contradiction.resolution (PREFER_GUARDIAN | MERGE | FLAG_AMBIGUOUS) is lossy. Consider aligning these types across learning.interfaces if detailed resolution tracking becomes important.

3. **Entity Matching:** ConsolidationService.consolidate() uses simple first-match by label for contradiction checking. In production, implement fuzzy matching or entity name comparison to find the correct existing node.

4. **Metrics Persistence:** LearningMetricsService holds metrics in memory (max 100). Full historical analysis requires TimescaleDB query. For long-running sessions, consider periodic flush to persistent storage.

5. **Drive State Access:** DRIVE_STATE_READER.getCurrentState() returns the last snapshot, not a fresh one. Timing-sensitive use cases may see stale drive data. Consider if fresher snapshots are needed via polling.
